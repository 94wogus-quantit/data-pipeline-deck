# 원시 데이터에서 연구 자산까지: Arkraft 데이터 카탈로그 구축 흐름

---

## 이 기능이 해결하려는 문제

독자적인 데이터베이스가 있다. 브로커의 거래 내역, 거래소의 틱 데이터, 펀드의 NAV 기록 — 연구에 쓰고 싶은 데이터인데, 막상 회귀분석 한 줄을 돌리기 전에 이미 몇 주치 작업이 쌓인다. 어떤 테이블이 있는지 파악하고, 각 컬럼이 실제로 무엇을 의미하는지 이해하고, 추출 로직을 짜고, 날짜 구멍을 처리하고, 포맷을 정규화하고, 동료들이 쓸 수 있도록 문서화까지.

Arkraft의 Data 기능은 바로 이 과정에서 겪는 부담을 줄이는 방향으로 개발되고 있다.

전통적인 금융 데이터 인프라는 전담 데이터 엔지니어링 인력을 필요로 하는 경우가 많다. ETL 파이프라인을 짜고 유지하는 사람, 각 데이터셋의 의미를 문서화하는 사람, 데이터 품질을 보장하는 사람, 최신 상태를 유지하는 사람. Arkraft는 그 역할들을 최대한 자동화하려는 방향으로 가고 있다. 데이터 배관 작업에 쓰는 시간이 줄고, 실제 알파를 만드는 일에 더 집중할 수 있게 되는 것 — 그게 이 기능이 궁극적으로 지향하는 바다.

# Zone 1. Data Source — 외부 DB를 연결하고 원하는 데이터를 고른다

## Scan — 데이터베이스의 구조를 읽는다

Scan의 설계 의도는 실제 데이터를 읽지 않고도 전체 DB 구조를 파악하는 것이다. information_schema 쿼리만 사용하기 때문에 full scan 없이 빠르게 완료되고, 수집한 메타데이터(테이블·컬럼·타입·인덱스·행 수 추정)가 이후 Propose 단계에서 AI가 테이블을 선별할 때 쓰는 판단 근거가 된다.

브로커나 거래소에서 온 데이터베이스는 테이블 이름이 `securities_master` 대신 `t_mst_sec`처럼 되어 있는 경우가 많다. 이름만으로는 의미를 파악하기 어렵기 때문에 컬럼 구조와 행 수까지 함께 수집해서 AI에 넘긴다. 실제 값 샘플링은 이후 Trial 단계에서 테이블별 3행을 직접 조회한다.

**앞으로 발전시키려는 방향** — 데이터 벤더로부터 받은 스펙 문서(데이터 정의서, 컬럼 설명서 등)를 Scan 단계에서 함께 업로드할 수 있게 하는 것을 고려하고 있다. 스키마 구조만 보는 것과, 벤더가 직접 작성한 컬럼 설명을 함께 참조하는 것은 AI의 이해도에서 큰 차이가 난다. 특히 `01 = 보통주, 02 = 우선주` 같은 코드 매핑이 문서에만 있는 경우에는 더욱 그렇다.

---

## Propose — AI가 분석에 쓸 만한 데이터셋 후보를 제안한다

Propose의 설계 의도는 수십~수백 개 테이블 중 연구에 실제로 쓸 만한 것만 선별하는 과정을 자동화하는 것이다. Scan 결과 전체를 AI(Claude)에 넘기면, AI가 도메인 지식을 적용해서 연구용 테이블과 운영용 테이블을 구분한다 — 감사 로그, 마이그레이션 히스토리, 세션 테이블은 걸러내고, 가격 시계열, 팩터 데이터, 기업행동 정보, 참조 데이터 중심으로 제안을 만든다.

현재 각 제안은 이런 형태로 나온다:

> **Daily OHLCV & Market Statistics (Korean Listed Stocks)**
>
> Daily OHLCV price, volume, and market statistics for Korean listed stocks, including price/volume adjustment factors for splits and dividends. Primary source for backtesting, return computation, and liquidity screening.
>
> **Tables**
> - `fnguide_kb.TS_STK_DAILY`
> - `fnguide_kb.TS_STOCK`
> - `fnguide_kb.TS_STK_ADJ_FACTOR`
> - `fnguide_kb.TS_STK_ADJ_FACTOR_DIVD`
>
> **Primary Key / Join 구조**
> Core grain: `STK_CD + TRD_DT` in `TS_STK_DAILY`. Join `TS_STOCK` on `STK_CD` for ISIN, CMP_CD, MKT_TYP metadata. Join `TS_STK_ADJ_FACTOR` on `STK_CD + TRD_DT` to apply cumulative price/volume adjustment factors. `TS_STK_ADJ_FACTOR_DIVD` provides dividend-only adjustment series.

스키마만 보고 이 정도 설명을 자동으로 만들어내는 구조다. 앞으로는 데이터 커버리지 기간, 유니버스 크기, 컬럼별 값 분포, 활용 가능한 유스케이스 같은 정보도 함께 제공하는 방향으로 발전시킬 계획이다.

**앞으로 발전시키려는 방향** — 지금은 AI가 스캔 결과를 보고 먼저 제안을 하는 방식인데, 앞으로는 사용자가 먼저 가이드를 주는 방향으로 바꿀 계획이다. "일별 수익률 시계열과 거래량 데이터가 필요해" 혹은 "종목별 재무 지표 관련 테이블만 봐줘" 같은 방향을 먼저 제시하면, AI가 그 맥락을 갖고 제안하는 구조다.

## Trial — 실제로 데이터를 뽑아보며 검증한다

Trial의 설계 의도는 전체 추출을 시작하기 전에 실제 데이터를 샘플로 검증해보는 것이다. 스키마 메타데이터만으로는 파악하기 어려운 것들 — null 비율, 값 분포, 날짜 공백 같은 실제 품질 지표 — 을 이 단계에서 먼저 확인하고 사용자가 판단할 수 있게 한다.

AI가 테이블별 스키마와 3행 샘플을 직접 조회한 뒤 extract_script.py를 작성하고, PostToolUse 훅이 즉시 자동 실행한다. 실패하면 stdout/stderr를 피드백으로 받아 수정을 반복하는 구조다. 결과로 나온 trial_results.csv와 품질 지표를 미리보기로 제공하고, 사용자가 Selected / Rejected를 결정한다.

**앞으로 발전시키려는 방향** — Trial 단계에서 사용자가 더 디테일한 가이드를 줄 수 있게 할 계획이다. "이 컬럼은 필요 없고, 이 기간 이후 데이터만 원해" 혹은 "수정주가 기준으로만 봐줘" 같은 방향을 전달하면, 시스템이 그 방향으로 다시 추출을 시도하는 구조다. 단순히 OK/NO를 고르는 것이 아니라, Trial 자체를 원하는 데이터셋을 찾아가는 과정으로 만드는 것이 목표다.

## Schedule — 동기화 주기를 설정한다

Schedule의 설계 의도는 데이터 갱신을 수동 개입 없이 자동화하는 것이다. 데이터셋을 선택한 뒤 동기화 주기(일별·주별·월별 등)를 설정하면, 이후 Incremental Extract가 해당 주기에 맞춰 자동으로 실행된다.

---

## Extract — 전체 히스토리를 날짜별로 쌓는다

Extract는 두 가지 모드로 실행된다.

**Full Extract** — Schedule 설정 후 최초 1회 실행. 전체 히스토리를 날짜 단위로 쪼개서 순차 추출한다 — 2020-01-01, 2020-01-02, 순서대로. 특정 날짜에 데이터가 없어도(공휴일, 거래소 폐장) 파이프라인 전체가 실패하지 않고 해당 날짜만 skip한다. 날짜별 파일들이 하나의 통합 Parquet 파일로 병합되어 S3에 올라가면 Zone 2 Pipeline이 자동으로 시작된다.

**Incremental Extract** — 이후 스케줄에 따라 반복 실행. trial의 extract_script.py를 재사용해 최근 7일 데이터를 추출하고 S3 기존 데이터와 diff를 계산한다 (new/updated). incremental parquet + diff.json을 S3에 올리면 Pipeline이 자동으로 트리거된다. Zone 2에서는 Detect-Normalize / CM Check / Spec을 skip하고, 최초 온보딩에서 만든 normalize_script.py + transform_script.py를 재사용해 증분 데이터를 처리한 뒤 Register로 카탈로그를 갱신한다.

대안 데이터나 독자 데이터에서 공백 없는 히스토리 시계열을 구축하는 과정 — 그 부분을 자동으로 처리하는 것이 Extract 단계의 목적이다.

---

# Zone 2. Pipeline — dataset 파일을 받아 분석 가능한 형태로 가공한다

Pipeline은 두 가지 경로에서 시작될 수 있다.

```
경로 A. Data Source  →  Extract 완료  →  S3에 Parquet 업로드  →  Pipeline 자동 시작
경로 B. 파일 직접 업로드  →  CSV / Parquet 업로드  →  Pipeline 자동 시작
```

실행 경로에 따라 처리 방식이 달라진다.

**최초 온보딩 (Full Extract 또는 파일 직접 업로드)**
Detect-Normalize → CM Check → Spec → Transform → Register 전 단계가 순서대로 실행된다. 대부분은 자동이지만, **Spec 단계에서 AI가 확신하지 못하는 부분은 사용자에게 질문하고 나서 다음으로 넘어가는 구조다.**

**운영 중 반복 (Incremental Extract)**
Incremental Extract가 최근 7일 데이터를 추출하고 기존 데이터와 diff를 계산해서 **incremental parquet + diff.json** 두 파일을 S3에 올리면, Pipeline이 자동으로 트리거된다. Detect-Normalize / CM Check / Spec은 skip하고, 최초 온보딩에서 만들어진 transform_script.py를 재사용해 증분 데이터를 처리한다. Pipeline은 diff.json을 참조해서 **new row는 append, updated row는 해당 row만 교체**하는 방식으로 기존 catalog parquet을 업데이트하고 Register로 카탈로그를 갱신한다. 사람 개입 없이 완전 자동으로 실행된다.

---

## Detect-Normalize — 데이터 구조를 파악하고 정규화한다

Detect-Normalize의 설계 의도는 입력 포맷에 상관없이 이후 파이프라인이 동일한 구조를 기대할 수 있도록 정규화하는 것이다. raw 파일을 읽어서 어떤 컬럼이 시간축인지, 엔티티 키인지, 값인지 판별하고 컬럼명을 정규화한다. 예를 들어 `STK_CD`는 종목 키, `TRD_DT`는 날짜, `CLOSE_PRC`는 값 컬럼으로 분류된다.

입력 파일이 CSV든 Parquet이든, 인코딩이나 구분자가 제각각이든 이 단계에서 통일된 형태로 정규화된다. 이 단계의 출력이 이후 모든 단계의 기반이 된다.

데이터 벤더로부터 받은 스펙 문서(데이터 정의서, 컬럼 설명서 등)를 이 단계에서 함께 업로드할 수 있다. AI가 스키마 구조만 보는 것과, 벤더가 직접 작성한 컬럼 설명을 함께 참조하는 것은 이해도에서 큰 차이가 난다 — 특히 `01 = 보통주, 02 = 우선주` 같은 코드 매핑이 문서에만 있는 경우에는 더욱 그렇다.

---

## CM Check — 데이터 품질을 검증한다

CM Check의 설계 의도는 AI 없이 결정론적으로 품질 게이트를 통과시키는 것이다. DuckDB로 수천만 행도 메모리 풀 로드 없이 처리한다.

총 6가지 항목을 체크한다:
- **파싱 가능 여부**: Parquet을 정상적으로 읽을 수 있는지
- **시간축 존재**: 날짜/타임스탬프 컬럼이 있는지
- **엔티티축 존재**: 종목 등 엔티티 식별 컬럼이 있는지 (없으면 경고)
- **값 컬럼 존재**: 숫자 또는 텍스트 값 컬럼이 있는지
- **데이터 완결성**: 전체 셀 대비 null 비율 — 50% 미만이면 실패, 50~80%는 경고
- **행 수**: 유효한 행이 존재하는지

결과는 세 단계다: **QUALIFIED**(통과) / **CONDITIONAL**(경고 있지만 진행) / **REJECTED**(중단). REJECTED가 아니면 Spec 생성으로 넘어간다.

---

## Spec 생성 — 카탈로그 메타데이터 스키마를 정의한다

Spec의 설계 의도는 데이터셋을 카탈로그에 등록하기 위한 메타데이터 스키마를 AI가 초안을 잡고, 불확실한 부분만 사용자에게 질문하는 구조로 만드는 것이다.

스펙에는 이런 것들이 담긴다:
- **cm_id**: `{Universe}.{Name}` 형식의 고유 식별자 (예: `KRX.DailyOHLCV`)
- **시간축 정의**: 이 데이터가 어떤 날짜/시간 단위인지
- **엔티티축 정의**: 종목별인지, 펀드별인지, 국가별인지
- **값 컬럼 메타데이터**: 각 값 컬럼의 단위, 타입, fill 방식 등

이 단계에서 사용자 응답이 필요한 순간은 총 네 가지다:

**① 불확실한 필드 질문**
AI가 특정 필드를 확신하지 못할 때 (신뢰도 < 0.8) 질문을 만든다. 예를 들어 "이 컬럼이 수정주가인지 원주가인지 확인해 주세요" 같은 식이다. 사용자가 답하면 AI가 그 답을 반영해서 스펙을 업데이트한다.

**② 카탈로그 충돌**
생성된 cm_id가 기존 카탈로그 항목과 겹치면 물어본다 — "건너뛸지, 이름을 바꿀지". 충돌이 없으면 자동으로 넘어간다.

**③ 엔티티 이름 입력**
데이터에 엔티티 컬럼이 명확하지 않은 경우(단일 엔티티) 사용자에게 직접 이름을 물어본다. 예: "이 데이터의 엔티티 식별자를 입력해 주세요 (예: ticker, stock_code)".

**④ ID 매칭 결과 선택**
엔티티 ID를 종목 마스터에 매칭했을 때 100% 매칭이 안 되면 어떻게 처리할지 물어본다:
- 매칭된 것만 등록
- 매칭된 것 + 미매칭은 raw 그대로 유지 (권장)
- 전체 raw로 등록
- 등록 취소

위 네 가지 중 해당하는 것이 없으면 Spec은 완전히 자동으로 완료된다.

---

## Transform — 분석에 쓰기 좋은 형태로 변환한다

Transform의 설계 의도는 Spec에 정의된 구조대로 데이터를 분석 툴에서 바로 쓸 수 있는 형태로 변환하는 것이다. 금융 데이터는 긴 형식(long-format)으로 되어 있는 경우가 많은데 — 종목, 날짜, 값이 하나의 행에 있는 구조 — 이걸 2D 평면 구조로 피벗한다.

값 컬럼마다 별도의 Parquet 파일이 만들어지고 카탈로그 경로에 저장된다. 만약 모든 값 컬럼이 이미 기존 카탈로그 항목에 매핑되는 경우라면 이 단계는 자동으로 건너뛴다.

---

## Register — 카탈로그에 등록한다

Register의 설계 의도는 AI 없이 결정론적으로 카탈로그 등록을 완료하는 것이다. spec.json과 transform.json을 읽어서 카탈로그 API(`/internal/data/register`)에 등록 요청을 보내고, 스펙에 정의된 대로 CatalogEntry가 생성된다.

등록이 완료되면 백그라운드에서 카탈로그 인덱스가 재빌드된다 — 전체 텍스트 검색을 위한 컬럼 목록을 갱신한다.

---

# Zone 3. Catalog — 팀의 공유 연구 자산으로 관리한다

카탈로그는 단순한 데이터 저장소가 아니다. 팀 전체가 데이터를 발견하고, 이해하고, 신뢰도를 쌓아가는 공간이다.

등록된 카탈로그 항목에는 이런 정보들이 담긴다:

| 항목 | 내용 |
|------|------|
| **유니버스** | 이 데이터가 커버하는 유니버스 (예: KRX, S&P 500) |
| **주기** | 일별, 틱, 월별 등 |
| **시간축** | 시계열의 정확한 날짜/시간 구조 |
| **엔티티축** | 종목별? 펀드별? 국가별? |
| **출처(Lineage)** | 어느 소스 데이터베이스에서 왔는지 |
| **컬럼 인덱스** | 전체 텍스트 검색 가능한 컬럼 목록 |

## 품질 등급: Research → Staging → Production

등록된 데이터가 바로 운영에 쓰이는 것이 아니다. 세 단계의 품질 등급을 거친다.

- **Research**: 탐색은 가능하지만 미검증 상태. 처음 등록된 데이터는 여기서 시작한다.
- **Staging**: 팀 리뷰를 거쳐 올라온 단계. 테스트 전략에 쓸 수 있다.
- **Production**: 라이브 전략에 써도 될 만큼 신뢰도가 검증된 상태.

카탈로그가 있으면 팀의 누구든 데이터셋을 검색하고, 무엇을 담고 있는지 이해하고, 어디서 왔는지 추적하고, 누가 쓰고 있는지 확인할 수 있다.

---

## 실제 사례로 보면

15년치 기관 주문 흐름 데이터를 가진 브로커의 독자 데이터베이스가 있다고 가정한다.

**Zone 1 — Data Source**
1. 접속 정보를 입력한다. Scan이 몇 분 안에 전체 스키마를 파악한다.
2. Propose가 60개 테이블 중 가치 있어 보이는 4개를 찾아낸다: 섹터별 일별 주문 흐름, 개별 거래 기록, 고객 활동 요약, 종목 참조 데이터.
3. Trial이 4개 모두에 실제 추출을 실행한다. 미리보기를 확인하다가 거래 기록 테이블에 2015년 이전 데이터 품질 문제가 있는 걸 발견한다. 나머지 3개는 선택하고 거래 기록은 거절한다.
4. 섹터별 주문 흐름을 매일 저녁 갱신하도록 설정한다.

**Zone 2 — Pipeline**

5. Full Extract가 15년치 데이터를 날짜별로 쌓아서 Parquet으로 저장한다.
6. Detect-Normalize → CM Check → Spec → Transform → Register가 자동으로 이어진다. Spec 단계에서 AI가 확신하지 못하는 필드가 있으면 질문이 올 수 있다 — 그 외에는 사람 개입 없이 완료된다.

**Zone 3 — Catalog**

7. 카탈로그에 등록된다. 일별 주기, KRX 유니버스로 태깅되고, 원본 브로커 데이터베이스까지의 lineage가 추적된다.
8. Research 상태로 시작해서, 팀 리뷰를 거쳐 Staging, Production으로 올라간다.

이제 동료들은 원본 데이터베이스나 파이프라인에 대해 아무것도 몰라도 이 데이터를 찾아서, 미리보고, 연구에 쓸 수 있다.
