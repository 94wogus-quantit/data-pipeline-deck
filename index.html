The Data Problem

The bottleneck in quant research is
getting reliable data, not running strategies.

Every new data source is a custom project: schema discovery, SQL authoring, quality negotiation, extraction scripts. A data engineer who understands both the source schema and the research domain is required — and scarce. The result: researchers wait weeks for data that should be available in days.

Onboarding a new data source
Manual: schema exploration, SQL, QA
Requires a data engineer who understands the source schema and the research domain. Each new source is a custom project — weeks of work before a researcher can use it.
→
Our design approach
AI interprets schema. One human decision.
AI scans schema, proposes research-relevant tables, writes and self-tests extraction scripts. The researcher reviews a data preview and decides which tables are worth extracting. No SQL. No manual pipeline authoring.
When the source database changes
Silent failures, broken backtests
Schema changes break extraction scripts. Stale data silently corrupts backtests. You discover the failure when your strategy stops making sense — not when the data broke.
→
Our design approach
Scheduled incremental syncs, re-checked quality
Incremental syncs are designed to append new data on a configured schedule. Quality gates re-run on each sync. The goal: source changes surface as pipeline warnings, not silent failures.
Data trust across the team
Everyone has their own version
Each researcher maintains personal copies of "the same" dataset with unknown cleaning differences. No shared confidence model — you don't know if a colleague's data is safe to use.
→
Our design approach
One catalog. Explicit, human-approved tiers.
Research → Staging → Production. Each promotion requires a deliberate decision. The tier tells the team what the data has been verified for — exploration, backtesting, or live strategy.
The core thesis: between a raw external database and a trusted research asset, there are exactly three decisions that require domain expertise — everything else is either deterministic or automatable. The pipeline is built around that constraint.

Core Design Argument

Why Not a Fully AI Pipeline?

The obvious path: let an AI model handle the entire pipeline end-to-end. We tried it. The problem: AI errors compound across steps — by step 5, you can't tell what went wrong or why. So we split the pipeline in two. AI handles the steps where context matters. Deterministic gates handle the steps where the rule can be stated precisely. The alternation is not a compromise — it's the design.

🤖 Claude (AI)
Steps with no clean rule
Used where the task requires domain expertise, semantic understanding, or resolution of genuine ambiguity that cannot be expressed as a rule.
Schema analysis — "daily_order_flow_by_sector" is research-relevant; "session_tokens" is operational noise. This requires domain knowledge, not a regex.
Column semantics — "adj_close" is a price field; "sedol" is an entity ID; "is_deleted" is a soft-delete flag. AI resolves this from context; no rule set covers every naming convention.
Entity dimension — Should this dataset be indexed by ticker, SEDOL, or ISIN? The answer depends on how the team plans to join it to other datasets. AI asks if unsure.
vs
⚙️ Deterministic Gate
Steps with a precise rule
Used where the standard can be stated precisely and must be enforced without exception. No AI — intentionally. Gate pass/fail is the audit trail.
CM Check (6 gates) — entity column present, date parseable, value numeric, null rate <threshold, no duplicate (entity, date) rows, minimum row count. Pass/fail, no ambiguity.
Register — catalog write is a contract: cm_id uniqueness enforced, schema validated, full lineage recorded. If CM Check didn't pass, Register cannot run. No exceptions.
Tier promotion — Research→Staging→Production requires explicit human approval. The system never auto-promotes. If Production data breaks, the approver is traceable.
Zone 2 Execution Sequence
AI · Detect-Normalize → Gate · CM Check → AI · Spec → AI · Transform → Det · Register
If a dataset passes CM Check, it has met 6 measurable criteria — regardless of which AI model ran, which person configured it, or when it ran. The gate result is the audit trail.
Why not Airflow + dbt?
Traditional ETL tools are authoring tools — a data engineer writes the pipeline. This approach fails when you don't have a data engineer who knows both the source schema and the research domain. Our approach: AI authors the interpretation steps (Detect-Normalize, Spec, Transform). Deterministic gates (CM Check, Register) verify the result. The authoring problem becomes an AI problem. The verification problem stays deterministic.

Zone 1 · Data Source

From Database Credentials to Validated Extract

Zone 1 is designed to prove the extraction is worth doing before committing to a full historical pull. AI handles schema analysis and script generation. You make one decision: is this data worth extracting? That's the only human step we could not design away.

01 LiveNo AI
Scan
Queries information_schema — every table, column, index, and row-count estimate. No actual row data downloaded. The metadata snapshot is the sole input for AI table selection.
02 LiveClaude
Propose
AI applies domain knowledge to surface research-relevant tables: price series, factor data, corporate actions, reference data. Operational tables — audit logs, sessions — filtered automatically.
03 LiveClaude
Trial
AI samples rows, writes extract_script.py, runs it via PostToolUse hook. Errors feed back as context. You review a preview — null rate, date coverage, value range — then mark each table Selected or Rejected.
04 LiveYou decide
Schedule
Set the sync cadence — daily, weekly, monthly. This is Touchpoint 1: the only decision you make in Zone 1. All downstream operations run on this interval automatically.
05 LiveClaude
Full Extract
Historical pull working day-by-day across the full date range. Empty dates skipped intelligently. Per-day Parquet merged via DuckDB streaming into extract_data.parquet, then Zone 2 triggered.
Recurring Live After onboarding, Incremental Extract runs on the configured schedule — reusing extract_script.py, fetching recent days, and triggering Zone 2. The design goal: no manual action needed for routine data updates.

Human Judgment Model

Three Irreducible Human Decisions

We started by asking: which steps genuinely require domain expertise, and which only feel like they do? The pipeline is designed so human judgment is required only where it cannot be substituted by rules or AI context — three specific moments, no more.

01
Zone 1 · After Trial
Review & Select Datasets
After AI runs trial extractions, you see a data preview: null rate, date coverage, value distribution. You decide which tables are worth extracting — Selected or Rejected.
Why human? AI can identify research-relevant schema patterns, but only a researcher knows if this specific dataset fits the strategy thesis and is worth the ongoing compute.
02
Zone 2 · During Spec — If Needed
Answer Spec Questions
If AI is uncertain about cm_id, entity dimension, or value semantics — it pauses and asks. If the spec is unambiguous, this touchpoint never occurs. Most runs skip it entirely.
Why human? "Is this adjusted or unadjusted close price?" is a financial domain question, not a data structure question. AI can detect ambiguity but not resolve it without context.
03
Zone 3 · Catalog
Approve Quality Upgrades
Manually promote datasets from Research → Staging → Production as confidence grows. Each upgrade is an explicit, traceable decision — the system never auto-promotes.
Why human? Live strategy data demands human accountability. If a Production dataset degrades, someone signed off on it — the system can page that person directly.
🔄
After the three initial decisions, the recurring sync loop — Incremental Extract → Zone 2 → Catalog update — is designed to run on the configured schedule without human intervention. This is the design target; the system surfaces alerts when something needs attention.

Design Thinking

Why This Architecture? Three Deliberate Choices.

Each design decision resolves a specific tension. Understanding them explains why the pipeline looks the way it does — and what we're still working out.

Decision 01 AI + Gates
Why alternate AI and deterministic gates?
Early version: fully AI-driven pipeline. Problem: errors compound across steps — by step 5, you can't tell what went wrong or why. Current design: AI handles interpretation, gates handle enforcement. If a dataset passes CM Check, it has provably met 6 measurable criteria — traceable and auditable, no matter which model ran.
Still working on: What happens when source data changes enough that a previously passing dataset now fails a gate? The re-validation pathway is being designed.
Decision 02 3 Touchpoints
Why exactly 3 human decisions, not 0?
We tried designing for zero human input. Three problems resisted automation: (1) Is this table worth the ongoing compute? — requires knowing the strategy thesis. (2) Is this field adjusted or unadjusted? — requires financial domain knowledge. (3) Is this data reliable enough for live strategies? — requires human accountability. These aren't automatable; they require judgment.
Open question: Touchpoint 2 (Spec disambiguation) is skipped most of the time. Can we reduce it further, or is occasional ambiguity unavoidable?
Decision 03 CM Format
Why a standardized 2D format for the catalog?
The catalog's value is joinability — datasets can be combined because they share a canonical form: a 2D date × entity Parquet matrix. This enables cross-dataset strategy building without ad-hoc alignment. Trade-off: some datasets don't fit cleanly (multi-dimensional data, irregular series). These either get forced into the model or can't be catalogued.
Still working on: How to handle datasets that are naturally multi-dimensional (e.g., options chains, order books). Force-fit to 2D loses information.
💬
These decisions shape the constraints we're building within. The architecture isn't finished — it's a working hypothesis being tested on real data sources. The design direction is toward a catalog that actively surfaces data quality issues before researchers encounter them, rather than waiting for a backtest to fail.
